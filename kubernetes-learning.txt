# Kubernetes (K8s) — Core Concepts Guide

---

## What is Kubernetes?

Kubernetes (K8s) is a **container orchestration tool** that manages containers 
across your infrastructure — whether that's a cloud provider, on-premises 
servers, or a hybrid of both.

At its core, Kubernetes solves three major problems for production applications:

| Problem | What K8s Does |
|---|---|
| **Downtime** | Keeps your app running continuously with no manual intervention |
| **Traffic spikes** | Scales containers up when load increases, scales down when it drops |
| **Disaster recovery** | Restores failed containers with data intact so nothing is lost |

---

## Core Components

### Pod

A **Pod** is the smallest deployable unit in Kubernetes. It acts as a 
wrapper that runs one or more containers — usually just one.

Key characteristics:
- Every Pod gets its own **IP address**
- Pods can communicate with each other using their IPs
- If a container inside a Pod crashes, Kubernetes automatically restarts it

**Example:**
```
Pod-1 (IP: 10.0.0.1)  ←──→  Pod-2 (IP: 10.0.0.2)
  └── Flask App                └── Database
```

> **Problem:** If a Pod crashes and gets replaced, the new Pod 
receives a *new* IP address. This breaks communication between Pods 
that relied on the old IP.

---

### Service

A **Service** solves the IP-change problem by assigning a **permanent, 
static IP address** to a Pod. If the Pod crashes and a new one starts, 
the Service ensures the new Pod gets the same IP as the old one — communication 
is never broken.

There are two types of Services:

- **Internal Service** — for components that should only be accessible *within* the cluster (e.g., a database). Users on the internet cannot reach it.
- **External Service** — for components that users interact with directly (e.g., a web frontend).

---

### Ingress

Even with an External Service, users would need to type something 
like `http://192.168.1.1:5000` to reach your app — which is not practical.

**Ingress** is the traffic manager of Kubernetes. It:
- Receives requests from the internet
- Routes them to the correct Service based on URL, path, or hostname
- Allows you to map a clean domain name (e.g., `www.myapp.com`) with no port number required

```
User types: www.myapp.com
     ↓
  [ Ingress ]   ←── routes by hostname/path
     ↓
  [ Service ]   ←── permanent IP
     ↓
   [ Pod ]      ←── your running app
```

---

### ConfigMap

Imagine your Flask app has the database URL hardcoded. If the database
 name ever changes, you'd have to:

1. Update the code
2. Rebuild the Docker image
3. Redeploy to Kubernetes

That's slow and error-prone.

**ConfigMap** stores **non-sensitive configuration data** (URLs, app 
settings, environment variables) *outside* the container image. To change 
a setting, you only update the ConfigMap — no rebuild, no redeploy.

> Use ConfigMap for: database URLs, feature flags, service endpoints, 
environment names.

---

### Secret

**Secret** works exactly like ConfigMap, but it's designed for **sensitive
 data** that must not be exposed:

- Passwords
- API keys
- Auth tokens
- TLS certificates

> Use Secret for anything you would not want committed to a public 
repository or seen in plain logs.

---

### Volume

A **Volume** attaches persistent storage to a Pod.

Without a Volume, all data stored inside a container is lost the moment 
that container or Pod crashes or restarts. With a Volume, the data persists
 independently of the Pod's lifecycle.

```
Without Volume:  Pod crashes → data gone ✗
With Volume:     Pod crashes → new Pod starts → data still there ✓
```

Volumes can point to:
- Local disk on the node
- Network storage (NFS, cloud block storage, etc.)

---

## How It All Fits Together

```
Internet
   ↓
[ Ingress ]          ← routes domain to the right service
   ↓
[ Service ]          ← permanent IP, load balancing
   ↓
[ Pod ]              ← runs your container(s)
   ├── ConfigMap     ← non-sensitive config injected at runtime
   ├── Secret        ← sensitive credentials injected securely
   └── Volume        ← persistent storage attached to the pod
```

---

## Quick Reference

| Component | Purpose | Stores sensitive data? |
|---|---|---|
| **Pod** | Smallest unit; runs containers | — |
| **Service** | Stable IP for Pod communication | — |
| **Ingress** | Routes external traffic by domain/path | — |
| **ConfigMap** | External non-sensitive configuration | ✗ No |
| **Secret** | External sensitive configuration | ✓ Yes |
| **Volume** | Persistent storage for Pod data | — |
